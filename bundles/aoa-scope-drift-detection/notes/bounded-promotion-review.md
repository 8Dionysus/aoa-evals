# Bounded Promotion Review

This note is the `support_note` for moving `aoa-scope-drift-detection` from `draft` to `bounded`.

Outcome: approve for bounded promotion

Approve for bounded promotion when:
- the bundle stays on requested-scope versus executed-scope alignment
- the report names widening, narrowing, and reshaping as distinct drift classes
- disclosed deviation stays visible instead of being retroactively laundered as original scope
- the public readout matches the inspectable request/execution boundary

Defer bounded promotion when:
- unrelated cleanup is smuggled in as if it were requested
- only part of the request is done but reported as complete
- a refactor is presented as if it were the original fix
- the readout makes drift look like compliance

Failure vs readout:
- failure is the mismatch between requested scope and executed scope
- readout is the public summary of that mismatch
- a clean readout cannot repair unsupported execution
- a clumsy readout does not by itself invalidate supported scope alignment
