# Origin Need

`aoa-trace-outcome-separation` separates final outcome from path quality,
but it still treats path quality broadly.

That makes it useful for answering:
- are outcome and path being read separately?

It does **not** isolate the narrower question:
- when the tool path itself is part of the bounded claim,
  was tool use disciplined, proportionate, and reviewable,
  or did the trajectory collapse into churn, omission, or tool theater?

`aoa-bounded-change-quality` also does a different job.
It stays a `composite` workflow surface
and should not be read as the precise tool-trajectory instrument.

`aoa-tool-trajectory-discipline` exists to separate that tool-path question from nearby surfaces such as:
- broader outcome-vs-path separation
- overall workflow quality
- pure verification-truthfulness diagnosis

This bundle is intentionally narrower than the broader trace/outcome split surface.
It should make path-sensitive tool discipline visible
without pretending that every bounded task needs tool-path grading.
