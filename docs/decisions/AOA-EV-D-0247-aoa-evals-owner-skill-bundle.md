## Context

AOA-EV-D-0246 removed 25 copied shared bundles and required any future
repository skill to earn an independent trigger, ABI, composition value,
coexistence behavior, and benefit over no skill. Manual work then compared
ordinary repository navigation with one provisional `aoa-evals` owner bundle.

Without the candidate, bounded questions could still be answered, but the
agent repeatedly reconstructed the same authority path across the chooser,
source bundle, Eval Forge, review guidance, dashboards, reports, and generated
readers. The candidate made that central-owner seam explicit and consistently
returned the source-first proof ceiling and a bounded disposition ABI.

The trials covered exact selection, readiness overread, a no-fit Eval Forge
handoff, unsupported maturity promotion, controlled generated-reader drift,
and a held-out owner-fit-versus-scope-drift selection. Negative cases covered
ordinary tests, an already selected local eval, generic analytics, raw session
retrieval, and decision lookup. Coexistence checks kept cross-repo selection
and local application with shared `aoa-eval`, durable rationale with
`aoa-decision`, and raw evidence retrieval with session-memory routing.

These trials support the owner-specific procedure and held-out transfer. They
do not establish token or latency improvement, cross-model or cross-host
parity, security, or general outcome superiority.

## Options Considered

- Keep no repository skill and reconstruct the central proof route each time.
- Add the procedure to shared `aoa-skills` or expand shared `aoa-eval` until it
  owns central proof interpretation.
- Advertise separate selection, review, and evolution skills.
- Admit one owner-local bundle with three internal modes and an exact derived
  repository projection.

## Decision

Admit one repository-owned `aoa-evals` bundle with internal `select`, `review`,
and `evolve` modes. Canonical source lives at `skills/aoa-evals/SKILL.md`.
`skills/port.manifest.json` declares version, lifecycle, visibility, admission,
and the exact repo-scoped Codex projection at `.agents/skills/aoa-evals`.

The bundle owns a callable route through central bounded-proof work, not proof
meaning itself. Source `EVAL.md`, `eval.yaml`, and admitted evidence remain
stronger than the skill, reports, readiness surfaces, receipts, generated
readers, MCP, KAG, and validation results.

Keep the three modes together until later held-out work proves independent
prompt-visible value. Do not require technique packages to invoke the skill.
Technique references inside an eval remain local lineage or support to inspect,
not inherited procedure dependencies.

## Rationale

The owner bundle reduces repeated route reconstruction while preserving the
boundary shared `aoa-eval` cannot own: central proof interpretation and owner
admission. One bundle limits prompt competition, shares a typed output ABI, and
lets a task assemble only the needed `select -> review -> evolve` edges rather
than forcing a fixed workflow.

The common `aoa-skills` home-port contract protects the durable source and
projection invariant. Manual trials, not that validator, justify admission.

## Consequences

- Positive: a fresh repository session discovers one evals-specific owner
  capability instead of copied shared bundles or no central front door.
- Positive: source/projection ownership, typed relations, effects, handoffs,
  failure dispositions, and health triggers are explicit.
- Positive: shared `aoa-eval`, decision, session-memory, stats, runtime, and
  proof owners keep their narrower authority.
- Tradeoff: one additional prompt-visible description consumes catalog space.
- Tradeoff: material model, workflow, or bundle changes require new manual
  routing and outcome review before lifecycle expansion.
- Follow-up: KAG may index the owner source and typed relations, but must mark
  `.agents/skills/aoa-evals` as a generated projection rather than a second
  authored skill.

## Boundaries

This decision does not admit a new eval bundle, change any proof claim,
category, maturity, baseline, report, or verdict, or turn the skill into proof
authority. It does not make manual session traces repository truth, require
siblings to create skill homes, or prove cross-model, cross-host, security,
latency, or token behavior.

## Validation

Admission uses the manually exercised cases summarized above, portable skill
shape review, exact home-port source/projection validation, fresh prompt-visible
inspection, decision-index parity, repo-local KAG regeneration, repository
validation, CI, and post-merge inspection. Structural checks protect only the
durable shape they actually cover.
