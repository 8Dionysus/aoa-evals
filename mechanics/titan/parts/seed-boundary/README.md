# Titan / Seed Boundary Part

## Role

This part owns the current seed-defined Titan canary source family.

It keeps named Titan, summon, memory, gate, runtime, bridge, and closeout
boundary pressure in one seed-defined part because the current artifacts share
one source shape and one validator route.

## Source Surfaces

- `mechanics/titan/parts/seed-boundary/seeds/titan*.yaml`
- `mechanics/titan/parts/seed-boundary/seeds/README.md`
- `mechanics/titan/parts/seed-boundary/seeds/AGENTS.md`
- `mechanics/titan/parts/seed-boundary/docs/TITAN_INCARNATION_CANARIES.md`
- `mechanics/titan/parts/seed-boundary/docs/TITAN_SUMMON_DISCIPLINE_CANARIES.md`
- `scripts/validate_repo.py` function `validate_titan_canary_surfaces`
- `tests/test_validate_repo.py` titan canary checks

## Route

`Titan boundary pressure -> seed canary YAML -> shape validation -> future executable scorer route -> bounded proof or owner handoff`

## Inputs

- Titan boundary pressure such as named identity, summon discipline, memory
  provenance, lineage non-erasure, runtime roster, bridge payload, gate
  boundary, or closeout receipt pressure;
- seed-defined `titan*.yaml` files with `id` or `eval_id` matching the filename
  stem;
- public-safe falsifiability through checks, required fields, expected result,
  forbidden output, or failure examples;
- seed-local route law plus incarnation and summon-discipline guide wording.

## Outputs

- validator-visible Titan seed canary shape;
- bounded seed-defined check records for future scorer conversion review;
- owner handoff posture when a seed cites stronger Titan role, memory, runtime,
  or summon truth;
- validation failures when a seed claims more than its seed-defined boundary can
  prove.

## Stronger Owner Split

`aoa-evals` owns the canary seed shape, seed-boundary part route, and bounded
proof posture.

`aoa-agents` owns Titan role classes, bearer identity, incarnation posture, and
summon boundary law. `aoa-memo` keeps memory authority. Runtime owners keep
activation, roster, and service truth.

## Stop-Lines

The seed family stays together because current canaries share one shape
contract and one validator route. Do not split named Titan, summon, memory,
gate, runtime, or bridge canaries into separate parts until a future slice adds
a distinct scorer, fixture, report, or validator contract.

This part is seed-defined only. It does not create full incarnation proof,
runtime activation, summon authority, memory sovereignty, mutation-gate bypass,
judgment-gate bypass, or executable scorer-backed proof.

Do not use canary presence as proof of a lawful summon, active Titan cohort,
remembered bearer incarnation, accepted memory write, hidden arena permission,
or owner-law change.
