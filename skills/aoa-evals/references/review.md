# Review bounded proof

## Required input

Require an exact source bundle or owner-routed proof object, the material
evidence/report/verdict/pressure, and enough repository, environment, baseline,
freshness, and review posture to interpret it. Route an unselected central need
to `select`.

## Procedure

1. Run the one declared command from `references/source-return.md` before any
   workspace or owner-path inspection. For an exact candidate packet ID, with
   or without an exact owner-review report path, this must be `review-context`;
   otherwise use `source`. Read the returned root owner `AGENTS.md`, then only
   the exact material paths returned by the packet or supplied beneath that
   owner root. Read-only evidence inspection is not an edit to those paths: do
   not enumerate or load their edit-route `AGENTS.md` cards. Do not use catalog
   search when the request already names the reviewed object. When citations
   are needed, read each exact document with line numbers on the first pass.
2. Read source `EVAL.md` and `eval.yaml` before any report, dashboard, receipt,
   MCP result, KAG node, or generated reader only when the selected object has
   an explicit accepted source-bundle link. When a candidate has no such link,
   keep that absence explicit; do not probe the skill package or substitute or
   search for a neighboring eval bundle. An unrelated source bundle cannot
   raise the candidate's proof state.
3. Read every material evidence ref and the actual named report or artifact.
   Preserve repository revision, model, host, tools, fixture, environment,
   baseline, accepted exits, scorer, and human review posture.
4. Distinguish each state explicitly:
   - source-contract readiness;
   - execution success;
   - evidence acceptance;
   - report validity;
   - bounded verdict;
   - public maturity;
   - central proof admission.

   No state implies the next.
5. For maturity, comparison, publication, or Eval Forge pressure, read
   `docs/guides/EVAL_FORGE_READINESS_LAYER.md` and the exact named owner-review
   report. Read implementation scripts only when their behavior is itself
   under review. Require the bundle-local evidence kind and explicit human
   confirmation named by root owner law. A green integrity sidecar cannot
   promote its neighbor. Do not inspect neighboring reports, dashboards,
   candidate reviews, or generated readers unless the named source or artifact
   cites one and it can materially change the disposition.
   Reading a readiness contract does not authorize its commands. Run a
   readiness check, route command, validator, generator, or test only when the
   request asks about current operational execution/readiness or when the
   source contract makes that result material to the bounded verdict. A
   contract-level proof-authority question normally needs no command execution.
6. Replay the strongest overread or evidence-gap directly from the inspected
   source and artifact. Once a missing source link, evidence acceptance,
   required human confirmation, or other gate makes every higher disposition
   impossible, stop expanding weaker surfaces. State which exact source or gate
   would change the disposition.
7. Return the narrowest supported review disposition and proof ceiling. Do not
   modify source, chooser, status, baseline, or generated readers.

## Claim limit

Review can establish only the bounded disposition supported by the inspected
source contract, accepted evidence, environment, scorer, artifact, and review
trail. Missing or weaker layers remain explicit.
